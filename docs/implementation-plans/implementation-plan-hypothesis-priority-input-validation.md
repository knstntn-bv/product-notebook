# План реализации: ввод приоритета гипотезы (валидация на blur, без автоподстановки дефолта)

## Обзор

**Требование:** [ESS-70](../../feature_requests/ESS-70%20Правка%20приоритета%20в%20гипотезах.md) (файл в `feature_requests/`).

**Проблема:** поле приоритета в редакторе гипотезы сейчас ведёт себя как контролируемый ввод с числом и подстановкой `?? 3` в `value`. При очистке поля состояние сразу отображается как дефолт, нельзя нормально стереть значение и ввести новое число.

**Цель:** свободное редактирование строки; проверка диапазона **1–99** на **потере фокуса** и при **сохранении**; при пустом или невалидном значении — **подсветка ошибки**, **сохранение заблокировано**; **закрытие диалога без сохранения** отменяет все правки (приоритет как до открытия).

**Область изменений:** клиент, `src/pages/HypothesesPage.tsx`. Схема БД и миграции **не** требуются.

**Связанный уже реализованный план:** [implementation-plan-hypothesis-priority-status-sort.md](./implementation-plan-hypothesis-priority-status-sort.md) (поле `priority` в таблице и базовый UI).

---

## Шаг 1: Общая функция разбора ввода

**Файл:** `src/pages/HypothesesPage.tsx` (или вынести в `src/lib/` при появлении второго потребителя).

**Поведение:**

- Вход: строка с инпута (как ввёл пользователь).
- Выход: дискриминированный результат, например:
  - `{ ok: true, value: number }` — целое в диапазоне **1–99**;
  - `{ ok: false }` — пустая строка, не число, не целое, вне диапазона.

**Замечание:** явно решить обработку пробелов (обычно `trim()` перед проверкой пустоты; либо считать пробелы невалидом — зафиксировать в коде один вариант и в чеклисте).

## Шаг 2: Состояние UI для поля Priority

Добавить рядом с логикой диалога редактирования гипотезы:

| Состояние | Назначение |
|-----------|------------|
| `priorityInput: string` | То, что показывает `<Input>`; единственный источник `value` для этого поля. |
| `priorityFieldError: boolean` (или строка сообщения) | Ошибка после blur / при попытке save. |

**Инициализация при открытии диалога:**

- **Создание:** например `priorityInput = "3"`, ошибка сброшена (дефолт согласован с продуктом и БД).
- **Редактирование:** `priorityInput = String(hypothesis.priority)` (или эквивалент с учётом `?? 3` только при **чтении** с сервера, не как подстановка в инпут при пустом черновике).

**Сброс при закрытии диалога без сохранения:** при `onOpenChange(false)`, Cancel, и любых путях, где `editingHypothesis` обнуляется — обнулять/сбрасывать и `priorityInput`, и `priorityFieldError`, чтобы следующее открытие снова брало данные из списка/БД.

---

## Шаг 3: Обработчики инпута

### 3.1 `onChange`

- Обновлять только `priorityInput` из `e.target.value`.
- По желанию UX: при вводе снимать `priorityFieldError` (ошибка снова появится после следующего blur или при save), чтобы поле не «залипало» красным, пока пользователь исправляет.

### 3.2 `onBlur`

- Вызвать общую функцию разбора для текущего `priorityInput`.
- Если **невалидно** → `priorityFieldError = true`, стили ошибки на поле.
- Если **валидно** → снять ошибку, записать число в `editingHypothesis.priority` (чтобы мутация сохранения и остальная логика опирались на согласованное число).

**Важно:** при невалидном blur **не** подставлять автоматически `3` и **не** молча менять отображаемую строку на дефолт — только флаг ошибки и подсветка.

---

## Шаг 4: Сохранение

- Перед вызовом `saveHypothesisMutation` (или внутри обработчика Save) ещё раз прогнать разбор **текущего** `priorityInput` (на случай, если пользователь не делал blur после правки).
- Если невалидно → выставить `priorityFieldError`, опционально toast, **не** вызывать API.
- Если валидно → передать в мутацию `priority: value` (для create/update как сейчас в `HypothesesPage.tsx`).

**Кнопка Save:**

- Либо `disabled`, пока `priorityFieldError` или пока текущий `priorityInput` не проходит разбор (второе покрывает «ни разу не было blur»).
- Либо оставить кнопку активной, но на клике выполнять ту же валидацию и выходить без save — команда может выбрать один вариант; рекомендация: disabled + визуально понятное состояние, либо активная кнопка с явным сообщением при клике.

---

## Шаг 5: Визуальная доступность

- На `<Input>` при ошибке: класс в духе `border-destructive` (согласовать с `components/ui/input` и существующими паттернами страницы).
- `aria-invalid={priorityFieldError}`.
- При необходимости: короткий текст ошибки под полем с `id` и `aria-describedby` на инпуте.

---

## Шаг 6: Закрытие диалога и откат

- Любое закрытие **без** успешного save должно приводить к сбросу локального состояния формы, включая `priorityInput` и ошибку. Тогда при следующем открытии записи приоритет снова тот, что в данных до попытки редактирования (из query/cache).
- Убедиться, что все ветки закрытия проходят через один и тот же сброс (крестик, overlay, Cancel, Esc — в зависимости от того, как устроен `EntityDialog` на этой странице).

---

## Шаг 7: Регрессия и чеклист ручной проверки

1. Редактирование: приоритет `7` → стереть всё → blur → красная обводка, save недоступен или не отправляет запрос.
2. Исправить на `8` → blur → ошибка снята, save сохраняет `8`.
3. Ввести `0`, `100`, `12a` → blur → ошибка.
4. Ввести валидное число, **без** blur нажать Save — поведение согласовано с выбранной стратегией (disabled до blur или валидация на клике).
5. С ошибкой на поле закрыть диалог → открыть ту же гипотезу → старый приоритет из БД.
6. Новая гипотеза: испортить приоритет, закрыть → создать снова → поле снова с дефолтным корректным состоянием.
7. Существующие сценарии сортировки по приоритету и таблица не ломаются (данные в БД по-прежнему только валидные числа).
