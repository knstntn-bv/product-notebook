# Полный экспорт / импорт контента продукта и связь с GitHub

**Статус:** зафиксированные продуктовые намерения (черновик для последующего обсуждения и анализа)

**Связь:** развитие пункта про двустороннюю синхронизацию с GitHub в `sync-ideas.md`.

---

## 1. Цель (долгосрочно)

Иметь возможность не только **выгрузить** весь контент продукта в формате, пригодном для Git, но и в перспективе **поддерживать двустороннюю синхронизацию** с репозиторием GitHub. На первом этапе сознательно упрощаем механику до **двух явных операций** без постоянного авто-синка.

---

## 2. Объём данных

- Экспортируется **полный срез контента продукта** (все сущности, которые команда включит в контракт выгрузки; конкретный перечень — на этапе спецификации).
- **Картирование файлов и идентичность записей** — **в том же духе, что уже зафиксировано для экспорта одной фичи:** UTF-8 Markdown, YAML frontmatter, **идентификация при импорте по полю `id` в frontmatter**, а не по имени файла на диске. Детали полей и шаблонов имён для остальных типов сущностей выводятся по аналогии с контрактом одиночного экспорта.

**Опора на существующий документ:** [2025-03-23-br-feature-export-single.md](./2025-03-23-br-feature-export-single.md) (формат `.md` + frontmatter + `id`).

---

## 3. Первая итерация: только кнопки, без непрерывной синхронизации

- Отдельное действие **«полный экспорт»** продукта (результат — набор файлов в согласованной структуре каталогов; способ доставки — см. §5).
- Отдельное действие **«полный импорт»** — восстановление/обновление данных продукта в приложении из того же формата.
- **Непрерывный** push/pull, webhooks и разрешение онлайн-конфликтов — **не** входят в первую итерацию; их можно наращивать позже на том же контракте файлов и `id`.

---

## 4. Импорт: правило слияния

Импорт выполняется как **слияние по `id`** (значение `id` из YAML frontmatter файла — ключ сопоставления с записью в продукте):

- если сущность с таким `id` **уже есть** — поля **обновляются** из импортированного файла;
- если такого `id` **нет** — создаётся **новая** запись;
- записи продукта, для которых в импортируемом наборе **нет** файла с соответствующим `id`, **не удаляются** и не изменяются этим импортом (импорт не выступает полной заменой всего содержимого продукта).

Отдельный режим «полная замена продукта» при необходимости можно добавить позже; в текущей фиксации — только слияние по `id`.

---

## 5. Ограничение: отдельного бэкенда нет

Сейчас предполагается только **веб-приложение** в рамках этого проекта (без отдельного сервиса-сервера). Для реального доступа к GitHub API из браузера это накладывает известные ограничения (секреты, OAuth, CORS).

Возможные траектории (не взаимоисключающие по этапам):

1. **MVP без GitHub в приложении:** полный экспорт как **скачивание архива** (или набора файлов); полный импорт как **загрузка** того же формата; коммиты и push пользователь делает **локально** в своём клоне репозитория. Контракт файлов и `id` уже готовит почву для API.
2. **Минимальный «бэкенд в том же репозитории»:** например **Supabase Edge Functions** — обмен кода OAuth на токен, вызовы GitHub API с серверным секретом, без отдельного деплоя «чужого» сервиса.
3. **Токен вручную (PAT):** пользователь вставляет Personal Access Token в настройках — проще в разработке, слабее по безопасности и UX; допустимо для внутреннего использования, если осознанно принято.

Выбор варианта для первой поставки — отдельное решение после оценки рисков и наличия Edge Functions в стеке.

---

## 6. Связь с будущей двусторонней синхронизацией

Когда появится синхронизация с GitHub «по живому», она опирается на **тот же** формат выгрузки и те же стабильные `id` в frontmatter; добавятся инкрементальные коммиты, подписка на события репозитория и политика расхождений. Кнопки полного экспорта и импорта остаются полезны для бэкапа, миграции и починки рассинхрона.
