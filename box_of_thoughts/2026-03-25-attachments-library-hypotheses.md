# Вложения (Attachments): библиотека файлов продукта и связь с гипотезами

**Статус:** зафиксированные продуктовые требования (черновик для передачи в анализ/реализацию)

**Контекст:** сущность для файлов, прикрепляемых к продукту; первая точка использования в UI — гипотезы. Один файл может быть связан с несколькими гипотезами.

---

## 1. Цель

- Пользователь может **загружать файлы** в контексте продукта и **прикреплять** уже загруженные к одной или нескольким гипотезам без повторной загрузки тех же байтов (через библиотеку).
- **Дедупликация при загрузке:** если содержимое уже есть в продукте, показать предупреждение в духе «Такой файл уже есть» и **предложить использовать** существующее вложение (вместо создания дубликата в хранилище).

---

## 2. Область и границы

| Параметр | Решение |
|----------|---------|
| Область дедупликации | Только внутри **одного продукта** (`product_id`). |
| Идентичность для дедупа | **Содержимое** (например криптографический хеш файла). Имя файла на диске пользователя **не** различает «другой» файл при совпадении байтов. |
| Разные файлы с одинаковым именем | Разные сущности, если содержимое разное. |
| После попадания в хранилище | Вложение — **одна сущность**; смена отображаемого имени у уже сохранённого файла — **то же** вложение, не новая загрузка. |
| Сеть | **Только онлайн** (без офлайн-сценариев на первом этапе). |

*Примечание для реализации:* при необходимости позже можно добавить явное действие «всё равно загрузить копию»; в текущей фиксации акцент на предложении переиспользовать существующее вложение.

---

## 3. Квоты и типы файлов

- **Один файл:** до **5 МБ**.
- **Суммарно по продукту:** до **100 МБ**.
- **Исполняемые** — **запрещены** (политика по расширениям/MIME — на усмотрение реализации, согласованно с безопасностью).

---

## 4. Жизненный цикл

| Событие | Поведение |
|---------|-----------|
| Удаление **гипотезы** | Вложения **не** удалять; связь с гипотезой убирается. В библиотеке/менеджере вложение помечается как **бесхозное** (ни к одной гипотезе не привязано), пока его не прикрепят снова или не удалят явно. |
| **Полное удаление** вложения | Допустимо: удаление сущности вложения (и объекта в объектном хранилище при отсутствии других ссылок — по правилам реализации). |
| Версионность | **Не** требуется. |

---

## 5. Клонирование гипотезы

- При **клонировании гипотезы** в рамках **того же продукта** связи с вложениями **по умолчанию сохраняются**: у новой гипотезы те же прикрепления, что и у исходной — **те же** сущности в библиотеке, **без** повторной загрузки и **без** дублирования объектов в Storage.
- Если позже в интерфейсе клонирования появятся опции (например «копия без вложений»), отключение переноса связей оформляется явным выбором пользователя; **базовый сценарий** остаётся — клон **с** сохранёнными вложениями.

---

## 6. Метаданные

- Расширенные поля (автор, комментарии, произвольные теги и т.п.) — **не** в scope.
- **Базовый технический набор** обязателен для квот, UI и политик: например связь с объектом в Storage, размер, тип/MIME или расширение, отображаемое имя в списке библиотеки.

---

## 7. UI / UX

- В **диалоге редактирования гипотезы** — кнопка, открывающая **диалог библиотеки** вложений продукта.
- В библиотеке: **загрузка новых** файлов и **прикрепление** существующих к текущей гипотезе.
- Отдельный раздел в навигации приложения **не обязателен** на первом этапе; достаточно модального/дополнительного диалога из редактора.
- **Превью** (картинки, PDF и т.д.) — желательно, но можно отложить на **вторую итерацию**; на первой — как минимум скачивание/открытие по ссылке.

---

## 8. Дедуп при загрузке (уточнение)

При выборе файла для загрузки, если хеш содержимого совпадает с уже существующим вложением этого продукта:

1. Показать предупреждение: файл с таким содержимым уже есть.
2. Предложить **использовать существующее** вложение (прикрепить к гипотезе без нового объекта в Storage).

---

## 9. Расширение на другие сущности

- Заложить возможность в будущем прикреплять те же вложения не только к гипотезам (фичи, цели и т.д.).
- Допустима **простая схема**: в таблице связей поля вроде типа сущности + идентификатор сущности (полиморфная связь), с пониманием компромисса: жёсткие FK на разные таблицы в Postgres так не выразить — целостность частично на стороне приложения. Альтернатива — отдельные junction-таблицы на тип; выбор за этапом проектирования схемы.
